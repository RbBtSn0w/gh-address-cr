## 2024-05-24 - Avoid dictionary `setdefault` in hot tight loops
**Learning:** In the telemetry reporting flow (`_error_prone_operations`), using `grouped.setdefault(..., {...})` dynamically instantiated a large new dictionary and `set()` on every single iteration just as a fallback. This caused massive allocation overhead processing tens of thousands of telemetry events.
**Action:** When grouping objects in tight loops, avoid `dict.setdefault()` if the default value requires instantiating complex types (like lists, dicts, or sets). Instead, use an explicit `if key not in dict: dict[key] = ...` to only pay the allocation cost once per unique key.

## 2024-05-24 - Avoid `any()` generators for performance-critical logic checks
**Learning:** Code in `src/gh_address_cr/core/logic_validation.py` (`_has_content`), `src/gh_address_cr/core/validation_evidence.py` (`validation_evidence_has_success`), and string validation (`char.isspace()`) was heavily utilizing `any()` generator expressions. The overhead of setting up the generator, especially in highly recursive functions operating on deeply nested dict structures, caused measurable delays compared to explicit `for` loops. Furthermore, redundant operations like checking `any(char.isspace())` when string values must strictly match space-free words are pure overhead.
**Action:** In performance-critical validation paths or recursive deep-checks, replace inline generators with explicit `for` loops and early `return True/False` statements. Never run expensive validations that are logically redundant due to stronger subsequent checks.
