# Runtime Kernel Durable Evidence Review Fix

## Context

- Feature: `018-runtime-kernel`
- Review finding: matching `command_executed` facts with `status: succeeded` but no durable evidence could complete `reply_thread` and `resolve_thread` work.
- Happy-path complexity assessment: adding ad hoc checks to terminal-state branches would make the main projection path more conditional. The first-principles boundary is a single trust filter for completion-worthy side-effect execution evidence.

## Fix

- Added a regression test for succeeded `reply_thread` and `resolve_thread` execution facts without `result_url`.
- Added centralized completion evidence filtering in `projections.py`.
- `reply_thread` and `resolve_thread` now require `status: succeeded` plus durable `result_url` before they count as completion evidence.
- Historical generation completion uses the same evidence filter, so stale unproven execution facts cannot create terminal history.
